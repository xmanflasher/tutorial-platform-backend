## 3. Collaboration Standards (開發規範)

- **Feature Development**: All new features must update SA/SD status.
  - **功能開發流程**：所有新功能須先在 `SA/SD` 進行影響分析與狀態更新。
- **Git Commit Workflow**: Apply Supervised Partitioned Git Commits using explicit prefix conventions (`feat:`, `fix:`, `chore:`, `docs:`).
  - **版本控制規範**：遵循語意化 Commit Message 與分區 Commit 策略。
- **Database Schema Sync**: Any change to Entity MUST also update the markdown ER Diagrams and `sql` import scripts.
  - **資料庫同步**：任何 Entity 的變動必須同步更新 Markdown ER 圖與 SQL 腳本。

## 4. Architecture & Refactoring Plan (專案架構與重構建議)

- **Dynamic Routing Guard (動態路由規範)**: <!-- id: ARCH-01 --> [Ref: SA-02, SD-01]
  For complex dynamic route groupings (e.g., `(main)` in `[slug]`), strictly apply `export const dynamic = 'force-dynamic'` to ensure consistent server-side resolution.
  針對嵌套路由組，必須強制標註動態渲染，以避免 Next.js 16/Turbopack 之路徑匹配衝突。
- **DTO Transformation (DTO 轉換規範)**: <!-- id: ARCH-02 --> [Ref: SD-01]
  Manual extraction of nested lazy-loaded entities (e.g., `LessonContent` inside `Gym`) is mandatory inside DTO builders to resolve payload gaps.
  手動提取巢狀延遲載入物件已成為標準轉換流程。
